# JWT Payload

Use custom JWT claims.

```go
type JwtClaims struct {
    UserID   uuid.UUID
    TenantID uuid.UUID
    RoleCode string
}
```

Reason:

- UserID identifies the current user.
- TenantID identifies the current workspace.
- RoleCode is enough for authorization middleware.

Do NOT store Password, Email, or other unnecessary information.

# Authentication API

## Login

```
POST /auth/login
```

Request

```json
{
    "email": "admin@school.com",
    "password": "password"
}
```

Response

```json
{
    "access_token": "...",
    "expires_in": 86400
}
```

---

## Profile

```
GET /auth/profile
```

Authorization

```
Bearer <token>
```

Response

```json
{
    "id": "...",
    "full_name": "...",
    "email": "...",
    "tenant": "...",
    "role": "SCHOOL_ADMIN"
}
```

# Authorization

Do NOT use Casbin.

Reason

Current Roles

- SUPER_ADMIN
- SCHOOL_ADMIN
- PARENT

Only three roles.

Permission-based authorization is unnecessary complexity.

---

# Middleware

Create middleware

```go
RequireRole(...)
```

Example

```go
router.POST(
    "/students",
    middleware.RequireRole(
        model.RoleSchoolAdmin,
    ),
)
```

Flow

```
JWT

↓

Read Claims

↓

RoleCode

↓

Allowed ?

↓

Handler
```

# Out of Scope

- Refresh Token
- Forgot Password
- Reset Password
- Register
- OAuth
- Email Verification
- OTP
- MFA

These belong to Version 2.

# Architecture Decision

Authentication is platform-level.

Every future module

- Payment
- Payroll
- Inventory
- Course Management
- Ledger

must reuse the same authentication layer.

Never duplicate authentication logic inside another module.
